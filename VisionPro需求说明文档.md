
## 1. 项目定位

这是一个基于 Web 技术的 3D 模型展示系统，目标设备包括：

- 普通 PC 浏览器
- 移动端浏览器
- Apple Vision Pro Safari

系统支持三种主要使用模式：

1. 普通浏览模式
2. AR/MR 模式
3. VR 模式

系统还带有一个管理后台，用于上传模型、缩略图和背景图。

需要特别说明的是，当前项目虽然已经具备基础功能，但整体项目架构、前端框架组织方式、部分技术方案仍然比较简单，稳定性和可维护性不足。因此后续开发不只是“继续在现有代码上叠功能”，还应把“升级为更稳定、更清晰、更易维护的项目架构与技术方案”作为明确目标之一。

要特别说明的一点是：

- 这不是一个原生 visionOS App。
- 它本质上是一个通过浏览器访问的 WebXR 项目。
- Vision Pro 与项目的“通信”方式，本质上是 `HTTPS + WebXR + WebSocket 日志桥`，不是自定义原生 SDK 通信。

## 2. 项目希望实现的核心效果

### 2.1 总体目标

用户进入页面后，应当可以：

- 浏览多个 3D 模型
- 查看模型名称、描述、缩略图
- 在普通浏览模式下旋转、缩放、切换模型
- 在 Vision Pro 中进入 VR 模式，以沉浸式方式查看模型
- 在 Vision Pro 中进入 AR/MR 模式，在真实环境背景中查看虚拟模型
- 通过管理后台上传模型和背景图，并在前台生效

### 2.2 适合本科生接手时的目标划分

建议把目标分成三层：

- 第一层：先让普通浏览模式和后台稳定可维护
- 第二层：再理解并维护 VR 模式
- 第三层：最后再深入 AR/MR 模式与 Vision Pro 交互细节

原因很简单：XR 部分的复杂度明显高于普通 Web 页面，不能让新手一开始就直接扎进 XR 手势和空间坐标问题。

### 2.3 新增中长期目标

除现有功能完善外，建议把以下两项作为后续明确目标写入项目范围：

1. 架构升级目标
   当前项目的架构、框架组织方式和技术实现相对简单，稳定性不足。后续应以“更稳定、更易维护、更利于多人协作”的工程结构为目标，评估并逐步替换当前较脆弱的组织方式与实现方案。
2. Demo 对齐目标
   目前你已经有一个“普通浏览模式下模型粒子加载效果”的 demo 项目。该 demo 可以作为普通浏览模式视觉效果和交互表现的参考目标，后续应评估如何把该粒子加载效果迁移、融合或重构到本项目中。

## 3. 技术栈与关键概念

### 3.1 前端技术

- `Three.js`
  - 用于搭建 3D 场景、相机、灯光、模型加载、基础渲染
- `WebXR`
  - 用于进入 Vision Pro 的 `immersive-vr` 和 `immersive-ar`
- `GaussianSplats3D`
  - 用于加载和渲染 `.ply / .splat / .ksplat / .spz` 这类高斯泼溅模型

### 3.2 后端技术

- `Node.js`
- `Express`
  - 提供静态资源服务和接口服务
- `multer`
  - 处理后台上传文件
- `ws`
  - 提供 WebSocket 日志桥，把前端日志打印到开发机终端

### 3.3 概念

- `GLB/GLTF`
  - 常规三角面网格模型格式，Three.js 原生支持较好
- `PLY`
  - 常用于点云或高斯泼溅源数据
- `SPLAT / KSPLAT / SPZ`
  - 高斯泼溅相关格式，其中 `KSPLAT` 更适合运行时加载
- `WebXR`
  - 浏览器中的 XR 标准接口
- `immersive-vr`
  - 沉浸式 VR 会话
- `immersive-ar`
  - 沉浸式 AR 会话
- `local-floor`
  - 以地面为参考的 XR 坐标系
- `transient-pointer`
  - Vision Pro 自然输入里常见的“瞬时指针”
- `gripSpace`
  - XR 输入源的抓取空间坐标
- `COOP/COEP`
  - 为了启用 `SharedArrayBuffer` 所需的跨源隔离策略
- `SharedArrayBuffer`
  - GaussianSplats3D 高性能路径依赖的浏览器能力

### 4.1 各模块职责

#### `index.html`

- 前台页面入口
- 定义展示画布、控制按钮、模型信息区、缩略图区
- 注入 `importmap`
- 初始化前端 WebSocket 日志桥

#### `js/app.js`

- 普通浏览模式主逻辑
- 初始化 Three.js 场景、相机、渲染器、轨道控制器
- 拉取模型列表和背景图列表
- 切换模型
- 按模型格式走不同加载逻辑
- 初始化 VR/AR 模块入口

#### `js/vr-module.js`

- Vision Pro 的核心技术文件
- 负责请求 XR 会话
- 负责 VR/MR 场景生命周期
- 负责沉浸式画廊、详情模式、手势交互、返回按钮、信息面板
- 负责在 XR 模式中正确集成 GaussianSplats3D

#### `js/plyViewer.js`

- 普通浏览模式下的高斯泼溅查看器
- 当模型是 `PLY/SPLAT/KSPLAT/SPZ` 时，前台会切到这个独立查看器
- 它会创建自己的容器并临时隐藏原始 Three.js canvas

#### `server.js`

- 启动 HTTPS 服务
- 提供静态资源服务
- 提供模型和背景图的增删改查 API
- 自动给前台页面加安全头
- 启动 WebSocket 日志服务
- 监听 `models/` 中新 PLY 文件并尝试自动转换为 `KSPLAT`

#### `admin/`

- 管理后台
- 上传模型、缩略图、背景图
- 编辑模型信息
- 删除模型
- 调整模型顺序
- 给模型绑定背景图

## 5. 系统整体流程

可以把整个系统分成 4 条主线：

1. 服务启动流程
2. 前台普通浏览流程
3. XR 模式流程
4. 管理后台流程

### 5.1 服务启动流程

```text
启动 server.js
  -> 检查目录与依赖
  -> 检查/加载 HTTPS 证书
  -> 设置 COOP/COEP/CSP 等安全头
  -> 启动静态资源服务
  -> 启动 API 服务
  -> 启动 WebSocket 日志桥
  -> 监听 models/ 目录中的 PLY 文件变化
```

### 5.2 前台普通浏览流程

```text
打开 index.html
  -> app.js init()
  -> 初始化 Three.js 场景
  -> 请求 /api/models
  -> 请求 /api/backgrounds
  -> 生成缩略图
  -> 自动加载第一个模型
  -> 初始化普通控制按钮
  -> 初始化 VR/AR 按钮状态
```

### 5.3 XR 模式流程

```text
点击 VR 或 MR 按钮
  -> VRModule 请求 XR Session
  -> 创建独立的 vrScene / vrCamera
  -> 初始化 XR 里的 splatViewer
  -> 再次从 /api/models 拉取模型列表
  -> 在空间中生成画廊
  -> 用户注视 + 捏合选中模型
  -> 进入详情模式
  -> 单手移动 / 双手缩放旋转
  -> 点击返回按钮回到画廊
  -> 结束会话后恢复普通浏览模式
```

### 5.4 管理后台流程

```text
打开 /admin/index.html
  -> 请求 /api/models
  -> 请求 /api/backgrounds
  -> 渲染列表
  -> 用户上传或编辑
  -> POST/PUT/DELETE 对应 API
  -> 写入 data/*.json
  -> 新文件保存到 models/ 或 images/
  -> 前台下次刷新后生效
```

## 6. 数据结构说明

### 6.1 模型数据 `data/models.json`

当前模型数据以 JSON 数组形式存储，每个模型大致包含：

```json
{
  "id": 1775009548022,
  "name": "模型名称",
  "description": "模型描述",
  "path": "/models/xxx.glb",
  "thumbnail": "/images/xxx.jpg",
  "format": "glb",
  "order": 1,
  "backgroundId": null,
  "createdAt": "2026-04-01T02:12:28.022Z"
}
```

字段含义：

- `id`: 模型唯一标识，当前由时间戳生成
- `name`: 模型名称
- `description`: 模型描述
- `path`: 模型文件访问路径
- `thumbnail`: 缩略图路径
- `format`: 模型格式
- `order`: 显示顺序
- `backgroundId`: 绑定的背景图 ID
- `createdAt`: 创建时间

### 6.2 背景图数据 `data/backgrounds.json`

背景图数据同样是 JSON 数组，典型结构：

```json
{
  "id": 1775000000000,
  "name": "背景图名称",
  "description": "",
  "path": "/images/xxx.jpg",
  "order": 1,
  "createdAt": "2026-04-01T00:00:00.000Z"
}
```

## 7. 普通浏览模式说明


### 7.1 入口

- 页面入口：`index.html`
- 主逻辑：`js/app.js`

### 7.2 页面组成

普通浏览模式包含：

- 中间的大画布区域
- 左右切换按钮
- 重置视角按钮
- 全屏按钮
- VR 按钮
- MR 按钮
- 模型名称和描述
- 底部缩略图列表

### 7.3 普通浏览模式的模型加载分流

普通浏览模式里，模型会分成两类：

#### A. 常规模型

包括：

- `glb`
- `gltf`
- `obj`
- `stl`
- `fbx`

处理方式：

- 用 Three.js 对应 Loader 加载
- 加载后做旋转、缩放、投影范围调整
- 加入主场景 `scene`
- 使用 `OrbitControls` 做旋转缩放观察

#### B. 高斯泼溅模型

包括：

- `ply`
- `splat`
- `ksplat`
- `spz`

处理方式：

- 清理普通模型
- 切换到 `plyViewer.js`
- 创建独立的 GaussianSplats3D Viewer
- 隐藏原来的 `canvas`
- 用独立 viewer 进行显示和交互

也就是说，普通浏览模式下实际上有两条渲染路径：

- Three.js 标准渲染路径
- GaussianSplats3D 独立查看器路径

### 7.4 普通浏览模式当前已实现的功能

- 模型列表读取
- 缩略图切换
- 左右切换模型
- 模型标题和描述显示
- 背景图切换
- 常规模型加载
- 高斯泼溅模型加载
- 相机重置
- 全屏切换
- 模型加载进度和错误提示

### 7.5 普通浏览模式应达到的效果

最低要求：

- 页面打开后能稳定显示第一个模型
- 用户可以点击缩略图切换模型
- 切换时旧模型被正确清理，不残留
- 不同格式模型都能按预期显示
- 重置视角按钮可靠可用
- 出错时有明显提示，而不是白屏

建议目标：

- 切换模型过程更平滑
- 背景图与模型匹配关系更清晰
- 统一不同格式模型的初始朝向与大小
- 对大模型给出更明确的加载状态

### 7.6 普通浏览模式的验收标准

- 至少能稳定展示一个 `GLB` 模型
- 至少能稳定展示一个 `PLY` 或 `KSPLAT` 模型
- 模型切换 10 次以上不崩溃
- 浏览器控制台不出现持续刷屏的报错
- 模型加载失败时页面仍可继续使用

## 8. VR 模式说明


### 8.1 VR 模式的本质

VR 模式不是把普通页面直接搬进 Vision Pro，而是：

- 重新请求一个 `immersive-vr` 会话
- 创建新的 `vrScene`
- 创建新的 `vrCamera`
- 在 XR 场景中重新组织模型和交互方式

### 8.2 VR 模式当前流程

进入 VR 后，系统会：

1. 请求 `immersive-vr`
2. 申请 `local-floor`
3. 可选申请 `hand-tracking` 与 `bounded-floor`
4. 创建 XR 独立场景
5. 重新读取模型列表
6. 把模型排成一个弧形空间画廊
7. 用户用 Vision Pro 的自然输入进行注视与捏合交互

### 8.3 VR 模式中的两种状态

#### 1. 画廊状态 `GALLERY`

特点：

- 多个模型排成弧形阵列
- 用户通过“看向模型”实现 hover 高亮
- 用户捏合一次即可进入详情模式

#### 2. 详情状态 `DETAIL`

特点：

- 只展示当前选中的模型
- 模型放在用户前方固定位置
- 出现返回按钮
- 出现模型说明面板
- 用户可以对模型做移动、缩放、旋转

### 8.4 VR 模式中的交互设计

#### 选择模型

- Vision Pro 的自然输入会触发 `transient-pointer`
- 系统先尝试从 `targetRayPose` 构造射线
- 如果拿不到有效矩阵，则退化为“XR 摄像机前向射线”

这是当前适配 Vision Pro 的关键兼容策略。

#### 单手移动模型

- 通过 `gripSpace` 获取手部姿态
- 记录上一帧和当前帧的 4x4 矩阵
- 用矩阵差值而不是简单坐标差实现移动

这么做是因为：

- Vision Pro 的输入源 `handedness` 可能不稳定
- 简单位置相减在父节点变换或用户移动时容易漂

#### 双手缩放和旋转

- 同时收集两个瞬时指针
- 通过两点距离变化计算缩放
- 通过两点夹角变化计算旋转

### 8.5 VR 模式当前已实现的效果

- 可以进入 VR 会话
- 可以在空间中看到模型画廊
- 可以 hover 高亮
- 可以选择模型进入详情模式
- 可以显示返回按钮与信息面板
- 可以单手平移
- 可以双手缩放和旋转
- 支持 GLB 和 Splat 模型

### 8.6 VR 模式希望达到的效果

最低要求：

- 能进入沉浸式 VR
- 模型画廊排列稳定
- 选中模型后能进入详情模式
- 返回按钮可靠
- 模型操作不明显漂移、不闪烁

建议目标：

- 模型 hover 更明显
- 详情模式 UI 更友好
- 模型缩放/旋转手感更自然
- 模型之间切换更快

### 8.7 VR 模式的验收标准

- Vision Pro 中可成功进入 VR
- 至少 3 个模型可在画廊中被看到
- 用户可选择模型并返回
- GLB 与 Splat 模型在 VR 中都能工作
- 高斯模型在 XR 中不出现明显双重渲染闪烁

## 9. AR/MR 模式说明

这里代码中按钮文案叫 `MR`，底层会话类型是 `immersive-ar`，因此可以理解为“AR/MR 模式”。

### 9.1 AR/MR 模式当前实现思路

AR/MR 模式与 VR 模式共用大量逻辑，但在视觉表现上不同：

- 使用 `immersive-ar`
- 场景背景设置为透明
- 渲染器 clear alpha 设为 `0`
- 利用 Vision Pro 的透视现实背景
- 添加地面对齐的阴影平面和遮挡平面

### 9.2 当前 AR/MR 模式能做什么

- 进入 MR 会话
- 在真实环境背景上叠加虚拟内容
- 使用和 VR 类似的画廊/详情切换逻辑
- 使用单手移动、双手缩放旋转

### 9.3 当前 AR/MR 模式还不是真正完整的“空间放置式 AR”

这一点非常重要：

- 当前代码没有实现真实世界平面检测后的“点按落位”
- 也没有实现命中测试 `hit-test` 式摆放
- 当前更接近“在透视现实背景中的固定空间画廊”

因此，不要误以为项目已经具备完整 AR 摆放系统。

### 9.4 AR/MR 模式想实现的效果

最低要求：

- 进入 MR 后背景透明，能看到现实环境
- 虚拟模型在地面参考系中稳定显示
- 模型能产生基本阴影效果
- 交互逻辑与 VR 模式一致

建议目标：

- 真正支持用户在现实空间中指定模型放置位置
- 支持更自然的模型落地和朝向调整
- 加强遮挡、阴影、深度关系的真实感

### 9.5 AR/MR 模式的验收标准

- Vision Pro 可进入 `immersive-ar`
- 模型不会整个贴在用户脸上或漂浮错位
- 切回普通模式后页面能恢复正常
- 至少一个 GLB 和一个 Splat 模型能在 MR 中显示

## 10. Vision Pro 通信、证书与访问说明


### 10.1 本项目与 Vision Pro 的通信本质

本项目不是用原生 SDK 与 Vision Pro 直接通信，而是：

- Vision Pro Safari 通过 `HTTPS` 访问网页
- 网页通过 `WebXR` 调用设备的 XR 能力
- 网页再通过 `WebSocket` 把日志发回 Node 服务端终端

也就是说：

- “Vision Pro 通信”本质是浏览器网络访问
- 核心协议是 `HTTPS/WSS`
- 核心前提是证书可信、网络可达、浏览器安全上下文满足要求

### 10.2 为什么必须要 HTTPS

原因有两个：

1. `WebXR` 要求安全上下文
2. `GaussianSplats3D` 的高性能路径依赖 `SharedArrayBuffer`，而这又要求严格安全头和安全上下文

### 10.3 当前证书体系

项目当前支持两种证书方案：

#### 方案 A：mkcert 证书，推荐

相关文件：

- `cert/mkcert-cert.pem`
- `cert/mkcert-key.pem`
- `cert/rootCA.pem`
- `cert/rootCA.cer`

特点：

- 本地开发更适合
- Vision Pro 更容易信任
- 当前 `server.js` 会优先读取这组证书

#### 方案 B：自签名证书，备用

相关文件：

- `cert/cert.pem`
- `cert/key.pem`

特点：

- 只适合临时开发测试
- 在 Vision Pro 上更容易遇到信任问题

### 10.4 正确的开发访问流程

推荐流程：

1. 在开发机安装 `mkcert`
2. 修改 `cert-config.json`
3. 保证 `subjectAltName.ip` 里包含开发机当前局域网 IP
4. 运行 `npm run cert`
5. 把 `cert/rootCA.cer` 安装到 Vision Pro
6. 在 Vision Pro 中对该根证书启用完全信任
7. 启动项目：`npm run dev`
8. 在 Vision Pro Safari 中打开 `https://开发机IP:3001`

### 10.5 为什么必须保证 IP 在证书里

Vision Pro 访问的是：

- `https://localhost:3001` 吗？不是
- 通常访问的是开发机在局域网里的 IP，例如 `https://10.x.x.x:3001`

所以证书必须覆盖：

- `localhost`
- `127.0.0.1`
- 当前机器在局域网中的真实 IP

否则即使服务能启动，设备端也会认为证书不匹配。

### 10.6 Vision Pro 上的访问要求

至少需要满足：

- Vision Pro 和开发机在同一网络，或者服务器可公网访问
- 3001 端口可达
- 证书被信任
- 页面通过 HTTPS 打开，不是 HTTP

### 10.7 WebSocket 日志桥的作用

Vision Pro 上调试网页比 PC 浏览器困难得多，因此项目额外做了一个日志桥：

- 前端劫持 `console.log/warn/error/info`
- 通过 `wss://当前主机` 发送回 Node 服务端
- 服务端打印到终端

这样即使不方便直接打开 Vision Pro DevTools，也能看到前端日志。

### 10.8 Vision Pro 开发中的常见失败点

- 证书未被信任
- 证书没有包含当前 IP
- 设备和开发机不在同一网段
- 使用了 HTTP 而不是 HTTPS
- 页面安全头不满足 `SharedArrayBuffer`
- 公司/校园网络拦截了某些端口或资源

#### 最推荐的常规模型格式

- 单文件
- 纹理和材质整合更方便
- Three.js 加载稳定
- 更适合部署

#### 最推荐的高斯泼溅格式

- `KSPLAT`

原因：

- 更适合运行时加载
- 比直接用原始 `PLY` 更适合面向展示系统

- 模型尽量以中心点为原点

- 模型比例尽量接近真实尺寸
- 纹理路径不要丢失
- 文件名不要包含太奇怪的特殊字符

### 11.5 缩略图要求

当前后台上传模型时要求必须上传缩略图。

建议：

- 使用正方形图
- 建议尺寸约 `200x200`

### 11.6 背景图要求

当前后台支持背景图格式：

- `jpg`
- `png`
- `gif`
- `webp`

背景图用于普通浏览模式的场景背景显示，不直接决定 XR 模式背景。

## 12. 模型加载方式和流程

这一部分是技术交接里最关键的一块。

### 12.1 模型列表获取

前台会先请求：

- `GET /api/models`

拿到所有模型的元数据后，再根据 `format` 决定如何加载。

### 12.2 普通模式下的常规模型加载流程

```text
用户选择模型
  -> app.js 读取 format
  -> 走 GLTF/OBJ/STL/FBX 对应 Loader
  -> 加载成功后做 normalizeAndOrientModel()
  -> 加上阴影属性
  -> 加入 scene
  -> resetCameraForModel()
```

### 12.3 普通模式下的高斯泼溅模型加载流程

```text
用户选择模型
  -> app.js 判断为 Gaussian Splat 格式
  -> 清理已有标准模型
  -> 调用 PlyViewer.cleanup()
  -> PlyViewer 创建独立 viewerContainer
  -> 隐藏原本 Three.js canvas
  -> GaussianSplats3D.Viewer.addSplatScene()
  -> viewer.start()
```

### 12.4 XR 模式下的高斯泼溅加载流程

XR 模式不能复用普通模式里的独立 viewer，因为 XR 下必须和 Three.js 共用渲染循环。

流程如下：

```text
进入 VR/MR
  -> VRModule.initSplatViewer()
  -> 使用 selfDrivenMode = false 创建 splatViewer
  -> splatViewer 挂接到 vrScene / vrCamera / renderer
  -> 在共享 XR 动画循环里只调用 splatViewer.update()
  -> 不额外调用 render()，避免双重渲染闪烁
```

### 12.5 PLY 转 KSPLAT 的服务端流程

服务端做了一个重要但目前还不完全闭环的能力：

```text
models/ 中出现新的 .ply
  -> fs.watch 监听到变化
  -> 加入转换队列
  -> 调用 gaussian-splats-3d 自带脚本
  -> 生成对应的 .ksplat 文件
```

### 12.6 当前转换链路的真实状态

当前情况是：

- 服务端会尝试生成 `.ksplat`
- 但 `models.json` 里仍然记录原始上传路径
- 前台普通模式和 XR 模式通常还是按原始 `path` 去加载

也就是说：

- “已经支持自动转换”
- 但“还没有真正做到前台优先使用转换后的 `KSPLAT`”

这属于后续优化的重要方向。

### 12.7 模型清理流程

切换模型或退出 XR 时，系统会尝试清理：

- 场景中的旧模型
- 几何体
- 材质
- 纹理
- splat scene
- XR 会话内的临时状态

这一步非常重要，因为：

- 3D 资源大
- Vision Pro 和浏览器 GPU 资源有限
- 如果不清理，容易出现内存上涨、卡顿、闪退、黑屏

## 13. 系统难点


### 13.1 Vision Pro 的 WebXR 交互不是普通教程那一套

普通 WebXR 示例经常默认：

- 有稳定 controller
- 有稳定 target ray
- 可以明显区分 left/right

但 Vision Pro 的自然输入更特殊：

- 常出现 `transient-pointer`
- `targetRayPose` 不一定稳定
- `handedness` 可能不是预期值

因此很多网上的 XR 教程不能直接照搬。

### 13.2 GaussianSplats3D 与 Three.js 在 XR 中集成难

高斯泼溅不是普通网格模型。

难点包括：

- 渲染器共享
- 相机共享
- XR 动画循环共享
- 避免双重 render
- 处理 `SharedArrayBuffer`
- 处理 worker、排序、显存占用

### 13.3 普通模式和 XR 模式的渲染路径不同

普通模式下：

- 高斯模型走独立 viewer

XR 模式下：

- 高斯模型必须挂进 XR 共享渲染循环

所以不能把“普通模式能显示”简单等价为“XR 模式一定能显示”。

### 13.4 AR/MR 的真实空间感没有想象中简单

AR/MR 不只是把背景透明：

- 还要考虑地面参考系
- 阴影
- 遮挡
- 模型放置点
- 手势稳定性
- 现实与虚拟的空间匹配

### 13.5 证书、网络和安全头是硬门槛

如果以下任意一项不对，XR 很可能直接不可用：

- HTTPS
- 证书信任
- COOP/COEP
- 访问地址与证书不匹配

优先级：

- 很高

### 14.2 PLY 自动转换为 KSPLAT 还没有形成闭环

现状：

- 服务端会生成 `.ksplat`
- 前台不一定会优先加载 `.ksplat`

影响：

- 自动转换的价值没有完全发挥
- 可能仍然在加载更重的原始 `PLY`

优先级：

- 很高

### 14.3 支持格式与后台上传入口不一致

现状：

- 前台代码支持 `OBJ/STL/FBX`
- 后台上传框没有开放这些格式

影响：

- 容易让维护者误解系统真实能力
- 文档与使用行为不统一

优先级：

- 中高

### 14.4 批量删除界面只给了部分格式选项

现状：

- 后台批量删除弹窗主要提供 `all` 和 `glb`
- 服务端本身其实支持按任意格式删除

影响：

- 前后端能力不一致

优先级：

- 中

### 14.5 模型顺序字段的语义不够严格

现状：

- `order` 字段存在
- 新增模型时不一定真正按 `order` 插入数组
- 当前更多是依赖数组顺序和后台重排接口

影响：

- 数据和展示顺序的含义不够清楚

优先级：

- 中

### 14.6 删除背景图后，没有同步清理模型里的 `backgroundId`

现状：

- 背景图被删后，模型记录里可能还保留旧 `backgroundId`

影响：

- 数据残留
- 前台虽然会回退为默认背景，但数据并不干净

优先级：

- 中

### 14.7 删除 PLY 时可能遗留自动生成的 KSPLAT 文件

现状：

- 删除模型接口主要删除 `model.path`
- 如果 PLY 转出了 `KSPLAT`，自动生成文件可能残留

影响：

- `models/` 目录会逐渐出现孤儿文件

优先级：

- 中高

### 14.8 `start.bat` 还偏向旧的自签名证书逻辑

现状：

- 当前系统更推荐 `mkcert`
- 但批处理脚本检查的是旧版 `key.pem`

影响：

- Windows 使用者容易误会当前推荐证书方案

优先级：

- 中

### 14.9 证书配置与日志里存在硬编码 IP

现状：

- `cert-config.json`
- `generate-mkcert.js`
- `server.js`

中都出现了 `10.110.161.70` 之类的固定 IP

影响：

- 换机器、换网络时容易出错
- 不够通用

优先级：

- 中高

### 14.10 管理后台依赖 CDN

现状：

- `admin/index.html` 引用了 BootCDN 的 Bootstrap 和图标
- 服务端对 `/admin` 页面使用了较宽松的 `COEP: unsafe-none`

影响：

- 离线环境或外网受限环境可能打不开样式
- 安全策略与前台不一致

优先级：

- 中

### 14.11 上传校验不足

现状：

- 后端主要按文件表单接收
- 没有很严格的格式、大小、安全校验

影响：

- 安全风险
- 错误文件可能被写入

优先级：

- 高

### 14.12 管理后台模型上传强制要求缩略图

现状：

- 上传模型必须同时传缩略图

影响：

- 流程稍重
- 自动化批量导入不方便

优先级：

- 中低

### 14.13 AR/MR 还不是完整平面摆放式 AR

现状：

- 目前更像“现实背景中的固定空间画廊”
- 没有真正的 `hit-test` 落位逻辑

影响：

- 如果老师或学生期待“把模型放到桌面上”，当前实现并不完整

优先级：

- 高，但建议放到第二阶段后再做

### 14.14 部分旧调试文档已经落后于当前代码

现状：

- 历史文档记录过 Vision Pro 射线问题
- 当前代码已经加入相机前向射线 fallback

影响：

- 新同学可能把旧问题当作当前现状

优先级：
